对当前项目工作区中**已修改但未提交**的代码进行逐项核对，定位潜在问题，输出结构化核对报告，作为提交/合并前的最后一道关卡。

## 任务目标

1. 识别当前工作区的所有修改（已暂存 + 未暂存 + 未跟踪）。
2. 逐文件分析变更内容，识别缺陷、规范违反、风险点。
3. 评估变更的影响范围（修改了哪些模块、可能影响哪些功能）。
4. 输出结构化核对报告，让用户决定是否可以提交。

## 执行步骤

1. **确认工作目录**：使用用户当前打开的项目根目录。
2. **查看变更概览**：
   - `git status --porcelain` 获取变更列表
   - `git diff --stat` 获取变更规模
   - 若工作区无变更：直接提示「当前工作区无代码修改，无需核对」并终止。
3. **读取变更内容**：
   - `git diff`（已修改未暂存） + `git diff --cached`（已暂存未提交）
   - 若 diff 输出过大（> 3000 行）：改用 `git diff --stat` + 按文件分批读取。
4. **逐文件核对**（按下方「核对维度」分析）。
5. **统计影响范围**：识别被修改文件所属的模块/功能，列出可能受影响的关联点。
6. **输出核对报告**：按指定格式输出，列出问题、风险与建议。

## 核对维度

### 1. 正确性

- 逻辑错误（边界条件、空值、类型转换）
- 死代码（修改后遗留的未使用变量/import/函数）
- 注释与代码不一致
- 接口调用与定义不匹配（参数个数、类型、顺序）

### 2. 规范性

- 命名风格（类/方法/变量）
- 缩进、空行、import 顺序
- 是否有明显格式化问题（不影响功能但与项目风格不一致）
- 公共方法/类是否缺少必要的注释

### 3. 完整性

- 修改后是否引入新的 TODO/FIXME
- 是否有未完成的占位符（如 `// TODO`、空方法体、`return null;`）
- 相关测试是否同步更新
- 文档（README/API 文档）是否需要同步

### 4. 风险性

- 是否修改了核心配置、数据库结构、权限校验
- 是否引入了新的外部依赖
- 是否存在性能退化风险（如循环嵌套、频繁 IO）
- 是否存在安全相关变更（鉴权、加密、SQL 注入、XSS）

### 5. 影响范围

- 受影响文件数、新增/删除行数
- 被修改的模块/类
- 可能受影响的上下游调用方
- 公共 API（Controller / Service 接口签名）是否变更

## 输出格式

```markdown
# 变更概览

- 变更文件数：X 个
- 新增行数：X 行
- 删除行数：X 行
- 受影响模块：模块A、模块B

# 必须修复（阻断提交）

1. [文件:行号] 问题描述 → 建议
2. ...

# 建议修复（不阻断但建议处理）

1. [文件:行号] 问题描述 → 建议
2. ...

# 风险提示

- 风险点 1：描述
- 风险点 2：描述

# 影响范围

- 修改文件：xxx、yyy
- 可能影响功能：功能A、功能B
- 公共 API 是否变更：是/否

# 核对结论

- [ ] 全部问题已修复
- [ ] 无阻断问题
- [ ] 风险已评估
- [ ] 可提交：是 / 否
```

## 场景处理

- **工作区无变更**：提示「当前工作区无代码修改，无需核对」并终止。
- **diff 输出过大**：使用 `git diff --stat` 概览 + 按文件单独读取完整 diff，避免上下文溢出。
- **非 Git 项目**：直接扫描工作区中被修改的文件（按 mtime 倒序或用户指定范围），同样按核对维度分析。
- **用户指定文件**：若用户在调用时明确指定了文件路径（如 `review-code src/main/java/XxxService.java`），仅核对指定文件。
- **无明显问题**：在「必须修复」和「建议修复」下写「无」，「核对结论」全部勾选，「可提交：是」。

## 注意事项

- 本命令只核对**修改的代码**，不评估未修改文件的整体质量。
- 「必须修复」仅放确实会引发 bug、回归或安全问题的项，不要把风格问题升级。
- 「建议修复」可放规范、可读性、潜在隐患等不阻断但值得改进的项。
- 若用户使用了用户规则中声明的"不解决安全问题/明文密码"等约定，核对时跳过对应项，避免误报。
